The Development Process

Stages — Defined

Stage 01
Requirements

Define every feature, page and function the system must have before any design or code begins. Requirements are split into functional and non-functional. Any new idea introduced in a later stage must be brought back to this document first before implementation.

  • Define project goals and target audience
  • List every page and its purpose
  • List every feature — functional and non-functional
  • Identify which content is static and which is dynamic (DB-driven)
  • Document approved and signed off before Stage 2 begins
EXIT CONDITION: Requirements document approved. No design begins until this stage is complete.
Stage 02
UI Design

Design every page visually in HTML/CSS. One page at a time. Each page is reviewed and approved before the next is designed. Design files are built with backend integration in mind — dynamic sections are clearly marked.

  • Design one page at a time
  • Each page reviewed and approved before moving to next
  • Mark all sections that will be fed by the database
  • Design changes are allowed but must be documented
  • All pages approved before Stage 3 begins
EXIT CONDITION: All pages designed and approved. No frontend code begins until this stage is complete.
Stage 03
Frontend Development

Build every page from the approved designs. Use static placeholder data where the database will later feed content. Code is written clean and structured for easy backend connection. One page built and approved before moving to the next.

  • Build from approved designs only — no improvising
  • Use placeholder data where DB will feed later
  • One page reviewed and approved before next page starts
  • Code written clean and structured for backend integration
  • All pages complete and approved before Stage 4 begins
EXIT CONDITION: All frontend pages complete, reviewed, and approved. No database design begins until this stage is complete.
Stage 04
Database Design

Design the database using the completed frontend as reference. Document every collection, every field and every relationship. Database design is approved before any backend code is written.

  • Use completed frontend to identify all dynamic data
  • Define every collection and its fields
  • Define relationships between collections
  • Document approved before Stage 5 begins
EXIT CONDITION: Database design document approved. No backend code begins until this stage is complete.
Stage 05
Backend Development

Build the server and API using the frontend and database design as reference. One endpoint at a time. Each endpoint tested independently before moving to the next. No changes to frontend during this stage.

  • Set up Node.js + Express server
  • Build API endpoints based on DB design
  • Test each endpoint independently before moving to next
  • No frontend changes during this stage
EXIT CONDITION: All API endpoints built and tested independently. No integration begins until this stage is complete.
Stage 06
Integration

Connect the frontend to the backend. Replace placeholder data with real API calls one section at a time. Test each connection immediately after making it.

  • Replace placeholder data section by section
  • Test each connection immediately after making it
  • Handle loading states and errors
  • Verify every page displays real data correctly
EXIT CONDITION: All sections connected to backend and verified working. No testing stage begins until this is complete.
Stage 07
Testing

Test the complete system — frontend, backend and database together. Every issue is logged, fixed and verified before deployment.

  • Functional testing — every feature works as expected
  • UI testing — every page correct on all screen sizes
  • Data testing — DB changes reflect on frontend correctly
  • Bug log created — every issue documented and fixed
EXIT CONDITION: All tests passed. Bug log cleared. No deployment until this stage is complete.
Stage 08
Deployment

Deploy frontend, backend and database separately. Verify everything works on the live environment. Create a maintenance document for updating content going forward.

  • Deploy frontend — Vercel or Netlify
  • Deploy backend — Railway or Render
  • Deploy database — MongoDB Atlas
  • Smoke test after deployment
  • Maintenance document created
EXIT CONDITION: Live site verified working. Maintenance document complete.
Process Rules

Ground Rules

  • 01Nothing moves to the next stage until the current stage is approved.
  • 02Every new idea or requirement change is documented in the requirements before implementation.
  • 03Each stage has a clear exit condition — the condition must be met before moving forward.
  • 04No assumptions — if something is unclear it is discussed and documented before building.
  • 05Every meaningful change to the project is committed to GitHub with a clear commit message.
  • 06This document is updated at every stage transition and whenever requirements change.
  • 07Design decisions are explained so the developer understands every choice made.
